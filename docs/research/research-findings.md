## AREA: er-core

MagicBlock Ephemeral Rollups (ERs) are SVM-compatible auxiliary validators that temporarily take over execution of specific Solana accounts. The lifecycle: a base-layer transaction "delegates" a program-owned PDA by transferring its owner to the Delegation Program (DELeGGvXpWV2fqJUhqcF5ZSYMS4JTLjteaAMARRSaeSh); the ER validator then executes transactions against that account at ~50ms slot times (marketing claims sub-10ms latency) with zero per-transaction fees on public nodes; state is committed back to Solana L1 periodically or on-demand via a program-side CPI (MagicIntentBundleBuilder.commit), and undelegation (commit_and_undelegate) returns ownership to the original program. Costs are concentrated at the edges: 0.0003 SOL per session at undelegation, 0.0001 SOL per commit, with 10 commits per delegation sponsored free by MagicBlock. Mainnet is live with regional endpoints (Asia/EU/US plus a TEE variant) alongside devnet equivalents and a "Magic Router" single endpoint (https://devnet-router.magicblock.app, wss supported) that auto-routes transactions to ER or base layer by inspecting writable accounts.

Code-level integration is light: Rust crate ephemeral-rollups-sdk (latest 0.15.4, published 2026-06-12 — actively maintained, releases weekly) with the "anchor" feature gives #[ephemeral] (on the program module), #[delegate] (delegation context; the PDA field uses AccountInfo with a `del` constraint and an auto-generated ctx.accounts.delegate_<field>() method taking DelegateConfig with optional validator pinning), #[commit] (ER-side commit/undelegate contexts), and #[action] (base-layer post-commit action handlers). Clients use a dual-connection pattern: base-layer RPC for init+delegate, ER RPC for everything on delegated accounts (with skipPreflight recommended); TS packages are @magicblock-labs/ephemeral-rollups-sdk (web3.js) and @magicblock-labs/ephemeral-rollups-kit (@solana/kit). Quickstart toolchain: Solana CLI 3.1.9, Rust 1.89.0, Anchor 1.0.2, Node 24.10.0.

The critical constraint for the Flash Trade question: program accounts are NEVER delegated — programs are cloned into the ER on demand, and EVERY Solana account is readable inside the ER, but ONLY delegated accounts can be written within an ER transaction. Since you do not control Flash Trade's program, its pool/custody/position accounts cannot be delegated, so an ER program CANNOT execute a state-changing CPI into Flash Trade from inside the ER. Reads of non-delegated accounts work via cloning (snapshot at request/transaction time — not continuously streamed, which is why MagicBlock ships a dedicated Pricing Oracle that pushes Pyth Lazer prices into ER-resident accounts at 50–200ms intervals, program ID PriCems5tHihc6UDXDjzjeawomAwBduWMGAi8ZUjppd). Two escape hatches back to L1: (1) Magic Actions — base-layer instructions scheduled inside an ER transaction via MagicIntentBundleBuilder.add_post_commit_actions([CallHandler{destination_program, accounts, args, escrow_authority, compute_units}]) that execute atomically on L1 right after the commit lands (docs say destination_program is "almost always your own crate::ID" and the target needs the #[action] macro, so arbitrary third-party targets are unverified); (2) an off-chain keeper that watches ER state via WebSocket and submits the Flash Trade transaction on L1. Cranks (MagicBlockInstruction::ScheduleTask via magicblock-magic-program-api) give on-ER scheduled execution at millisecond intervals with no external infrastructure — ideal for continuously evaluating trailing-stop/OCO trigger conditions against the oracle feed. The docs' own listed crank use cases include "periodic price updates... portfolio rebalancing, and liquidation checks," which is essentially direction A.

FEASIBILITY: Direction A (advanced order engine for Flash Trade with trigger logic in ER) is architecturally sound and the best fit for the judging criteria, with one load-bearing caveat: order EXECUTION on Flash Trade cannot happen inside the ER (only delegated accounts are writable; Flash Trade's accounts can't be delegated). The proven-by-docs architecture is: (1) order PDAs (trailing stop / OCO state) owned by your Anchor program, delegated to the ER; (2) MagicBlock's real-time Pyth Lazer oracle accounts inside the ER (50-200ms updates, program PriCems5tHihc6UDXDjzjeawomAwBduWMGAi8ZUjppd) as the price source; (3) a crank scheduled at e.g. 100ms intervals evaluating trigger conditions fully on-chain inside the ER (zero fees, so millions of evaluation ticks cost nothing — this is the 'load-bearing ER' story judges want); (4) on trigger, commit_and_undelegate plus either a Magic Action into your own L1 program that CPIs into Flash Trade, or a thin keeper bot subscribed to the ER WebSocket that fires the Flash Trade transaction on L1. Hour-zero derisking tasks: (a) verify on devnet that a Magic Action can drive a CPI into an external program from your own #[action] handler, (b) confirm oracle asset coverage and crank minimum interval, (c) confirm Flash Trade devnet availability. Budget realities are friendly: devnet ER is free, commits cost 0.0001 SOL with 10 sponsored per delegation, and the dual-connection client pattern plus quickstart counter template means the ER skeleton can be running in under an hour (there is also a local 'magicblock' Claude skill at C:\\Users\\musha\\.claude\\skills\\magicblock with current SDK 0.11+ MagicIntentBundleBuilder patterns, crank code, fee-vault wiring, and Magic Action gotchas — use it during the build). Direction C (PvP arena) is also well supported (ephemeral accounts for session state, VRF free inside ER (<50ms), session keys for signature-less UX) but is closer to MagicBlock's own game templates, so it scores lower on creativity; direction B (copy trading) suffers from the same constraint as A (the copied execution must happen on L1 via keeper), with less ER-native logic to show off. Recommend A, optionally borrowing session keys + Magic Router from C's toolbox for a polished one-click trading UX.

## AREA: er-plugins

MagicBlock's plugin stack is real and surprisingly deep for the trading theme. (1) Private ERs (PERs) run a full SVM ephemeral rollup inside an Intel TDX TEE. You delegate an account to the TEE validator (mainnet-tee.magicblock.app / devnet-tee.magicblock.app), then create an `EphemeralPermission` account ON the rollup (paid by the delegated PDA, via Permission Program ACLseoPoyC3cBqoUtkbjZ4aDrkurZW86v19pXz2XQnp1, ephemeral-rollups-sdk v0.14+). Privacy is opt-in per account (`set_privacy(is_private)`) with member lists and bitmask flags (AUTHORITY, TX_LOGS, TX_BALANCES, TX_MESSAGE, ACCOUNT_SIGNATURES). Reads against the TEE RPC require an auth token obtained by signing a challenge (`https://devnet-tee.magicblock.app?token={authToken}`), and the TEE validator checks the requester against the permission account. So yes — hidden stop-loss levels / dark orders are plausible TODAY: the order account's state executes inside the TEE and unauthorized parties can't query logs/messages/balances, on both devnet and mainnet TEE endpoints. Big caveat: the docs never explicitly state that raw account DATA (getAccountInfo) is hidden from non-members (flags cover logs/balances/messages/signatures), and committed state becomes public on the base layer when you commit/undelegate — design so secrets are only revealed after execution. Separately, MagicBlock already ships a hosted "Private Payments API" (payments.magicblock.app) with /quote and /swap endpoints: public mode is a pure Jupiter/Metis pass-through; private mode routes swap output into a program-owned account and schedules a delayed (0–600s), split (1–14 parts) private transfer via a one-shot "Hydra crank" — essentially dark-pool-style swaps as a turnkey API (mainnet).

(2) VRF: `ephemeral_vrf_sdk` (crate v0.3.0, features=["anchor"]) gives a request-and-callback pattern (CPI request with caller_seed + callback discriminator; oracle fulfills; VRF program verifies an RFC 9381 Ristretto/Schnorr proof and CPIs your callback, identity-checked via VRF_PROGRAM_IDENTITY). Latency: <50ms and FREE inside an ER; on base Solana 0.0008 SOL (<500ms) or 0.0005 SOL (1–2s) per fulfilled request. Works on devnet and localnet (roll-dice example + live demo).

(3) Session keys: two parts — client SDK `@magicblock-labs/gum-react-sdk` (React hook `useSessionKeyManager` exposing createSession({targetProgram, topUpLamports, validUntil}), revokeSession, signAndSendTransaction) and Anchor crate `session-keys = "1.0.0"` (derive `Session`, `#[session(signer=..., authority=...)]`, `#[session_auth_or(...)]` guard; program ID KeyspM2ssCJbqUhQ4k7sveSiY4WjnYsrXkC8oDbwde5). An ephemeral keypair lives client-side; a SessionToken PDA encodes expiry+scope; `topUpLamports` pre-funds the session key so the user gets zero popups during the session — exactly the real-time trading UX needed. Note ER transactions are gasless anyway, so session keys matter most for base-layer txs or you can just use a local ephemeral key for ER txs.

(4) Automation/cranks: YES, native. The ER validator itself executes scheduled tasks — your program CPIs `ScheduleTask` into the Magic program with task_id, execution_interval_millis, iterations, and a serialized instruction list; the validator then fires it on schedule with no external keeper and no per-tick client transaction (crank-counter example confirms). Account must be delegated first. This is the load-bearing primitive for direction A: a trigger-checking instruction (read price oracle account in ER, compare to stop/trail level, fire) can run every N ms inside the ER for free. Combined with Magic Actions (`MagicIntentBundleBuilder` + `#[action]`/`#[commit]` macros: attach post-commit CallHandler instructions that execute atomically on the base layer right after an ER commit, ~200k CU each, sequential), you get the full loop: crank watches price in ER → condition met → commit + post-commit action CPIs a base-layer program (e.g., Flash Trade order placement) using freshly committed state. MagicBlock also has a real-time price oracle feed usable inside ERs (tools/oracle, real-time-price-feed template), giving you live prices for trigger logic without an external feeder. ER pricing: 0 per-tx fees; 0.0003 SOL per session close + 0.0001 SOL per commit.

The canonical on-chain pattern (anchor-counter/programs/public-counter/src/lib.rs) is: `#[ephemeral]` on the program module, a `#[delegate]` accounts struct with `#[account(mut, del)]` giving you a generated `ctx.accounts.delegate_pda(payer, seeds, DelegateConfig { validator, .. })` CPI, and a `#[commit]` accounts struct whose handlers call `MagicIntentBundleBuilder::new(payer, magic_context, magic_program).commit(&[acct])` or `.commit_and_undelegate(&[acct]).build_and_invoke()`. Anchor state must be flushed with `counter.exit(&crate::ID)?` before committing inside the same ix.

Three examples are directly load-bearing for an order-trigger engine. (1) crank-counter: schedules fully on-chain recurring execution inside the ER by CPI-ing `MagicBlockInstruction::ScheduleTask(ScheduleTaskArgs { task_id, execution_interval_millis, iterations, instructions })` to MAGIC_PROGRAM_ID — i.e., a keeper/crank with millisecond intervals and NO off-chain bot. (2) magic-actions: post-commit handlers — `MagicIntentBundleBuilder...commit(&[...]).add_post_commit_actions([CallHandler { destination_program, accounts, args, escrow_authority, compute_units }])` runs a base-layer instruction automatically when the ER commit lands (their example: ER counter commit triggers base-layer leaderboard update). This is exactly the "trigger fires in ER → settle/execute on base layer" bridge a trading product needs. (3) real-time-pricing-oracle (separate repo, also surfaced as the docs "Real-Time Price Feed" template and starter-kits/oracle-demo): a Rust chain-pusher streams Pyth Lazer / Stork prices into ER-resident PDAs under program `PriCems5tHihc6UDXDjzjeawomAwBduWMGAi8ZUjppd` (seeds ["price_feed", "pyth-lazer", feed_id]; i64 price at byte offset 73, scale by 10^exponent), with live devnet-ER feed addresses for SOL/BTC/ETH/USDC and an on-chain consumer example using PriceUpdateV2 + get_price_no_older_than.

Frontend: two stacks exist. anchor-counter/app and session-keys/app are CRA (craco) React apps using @solana/wallet-adapter-react (WalletMultiButton) for the base layer; for ER transactions they derive a temp keypair from the wallet pubkey (`Keypair.fromSeed(publicKey.toBytes())`), sign locally, and `sendRawTransaction(..., {skipPreflight:true})` to a second `Connection` pointed at the ER — so no wallet popup per ER tx, and they confirm at 'processed' commitment (~30–100 ms, measured in-app). Both layers get separate `onAccountChange` WS subscriptions so the UI shows base vs ephemeral state side by side. The roll-dice/app frontend is the modern skeleton: Next.js 15 + React 19 + Tailwind + shadcn/ui + @magicblock-labs/ephemeral-rollups-sdk 0.14.3, and uses `ConnectionMagicRouter` with `getDelegationStatus(pda)` to discover which ER validator (FQDN) holds the delegated account, then opens a direct Connection to that validator. Endpoints are hardcoded in rewards-delegated-vrf/dashboard/lib/endpoints.ts: base via rpc.magicblock.app/devnet|mainnet; ERs at devnet-as/devnet-us.magicblock.app and mainnet as.magicblock.app / us.magicblock.app. A code comment warns: the router (devnet.magicblock.app) proxies HTTP per-request but WS subscriptions bind to one ER, so frontends should pin a regional ER endpoint for subscriptions.

Best starting skeleton for an order-trigger engine + Next.js UI: program = anchor-counter structure, merged with magic-actions (post-commit base-layer execution) and crank-counter (ScheduleTask loop evaluating trigger conditions against the ephemeral-oracle Pyth Lazer PDA each N ms); frontend = fork roll-dice/app (Next.js + shadcn + ConnectionMagicRouter dual-connection already wired, IDL-driven Anchor client). Architecture that the examples directly support: user creates Order PDA on base layer → delegate to ER → crank evaluates price PDA vs trigger every ~100ms inside ER → on trigger, commit_and_undelegate the order with a post-commit Magic Action (or emit for an off-chain executor) that performs the Flash Trade instruction on the base layer. The repo also ships test-locally.sh which boots a full local cluster (base validator + ER + query-filtering service + VRF oracle) for offline development, and rock-paper-scissor gives a complete PvP-with-hidden-state pattern (plus Next-style app with link/QR join) if direction C is chosen.

FEASIBILITY: Direction A (advanced order engine: trailing stops / OCO with trigger logic in the ER) is the best-supported build and can be assembled almost entirely from existing, working pieces: (1) Order PDA program cloned from anchor-counter's delegate/commit skeleton (~150 lines gets you delegation lifecycle); (2) price input is FREE — MagicBlock already runs a Pyth Lazer oracle pushing SOL/BTC/ETH/USDC prices into ER PDAs at a known address with a documented 73-byte-offset decode, so 'real data, no mocks' is satisfied on day one; (3) the crank (ScheduleTask) gives you an ON-CHAIN keeper that re-evaluates trigger conditions every N milliseconds inside the ER — a genuinely novel demo point ('your stop-loss is checked every 100ms on-chain, not by a bot'); (4) Magic Actions gives the settlement bridge: when a trigger fires, commit the order and automatically run a base-layer instruction. ## AREA: er-oracles

YES — there is a first-party, supported, production-grade way to read live market prices INSIDE a MagicBlock Ephemeral Rollup, and I verified it working live on both devnet ER and mainnet ER today (2026-06-12). MagicBlock ships an "ephemeral oracle" (repo: magicblock-labs/real-time-pricing-oracle, program ID PriCems5tHihc6UDXDjzjeawomAwBduWMGAi8ZUjppd) whose chain pusher ingests Pyth Lazer (and Stork) feeds and writes them into PDA accounts that are DELEGATED into the Ephemeral Rollup via the delegation program (DELeGGvXpWV2fqJUhqcF5ZSYMS4JTLjteaAMARRSaeSh). Docs state 50–200 ms update intervals depending on asset (most crypto feeds are fixed_rate@50ms in pyth_lazer_list.json). I empirically confirmed sub-second updates: three reads of the BTC/USD feed on mainnet ER ~300 ms apart returned three different prices, and the SOL/USD feed's publish_time was current on both clusters. There are 184 feed accounts on devnet ER and 135 on mainnet ER; the Lazer list has 125 symbols (BTC, ETH, SOL, USDC, USDT, PEPE, BONK, etc.).

Integration is trivially simple and confirmed against source code. Feed PDA = findProgramAddress(["price_feed", provider, symbol], PriCems5...) where provider is "pyth-lazer" and symbol is the Pyth Lazer feed ID as a DECIMAL STRING ("1"=BTCUSD, "2"=ETHUSD, "6"=SOLUSD, "7"=USDCUSD — I derived these locally and they exactly match the documented live addresses, e.g. SOL/USD = ENYwebBThHzmzwPLAQvCucUTsjyfBSZdD9ViXksS4jPu). The 134-byte account is a "PriceUpdateV3" struct deliberately binary-compatible with Pyth's PriceUpdateV2: your ER program just does `cargo add pyth_solana_receiver_sdk`, takes the feed as an unchecked AccountInfo, calls `PriceUpdateV2::try_deserialize_unchecked(...)` then `get_price_no_older_than(&Clock::get()?, max_age, &feed_account_pubkey.to_bytes())` (feed_id == the account address). Raw layout for clients: price is an i64 at byte offset 73, scaled by 10^exponent (crypto feeds exponent -8); subscribe via plain `accountSubscribe` over wss://devnet.magicblock.app or wss://mainnet.magicblock.app. Only MagicBlock's oracle identity (MPUxHCpNUy3K1CSVhebAmTbcTCKVxfk9YMDcUP2ZnEA) can write updates; the pusher is open source but running your own requires a Pyth Lazer auth token.

For the trailing-stop / advanced-order engine (direction A), this is exactly the intended architecture and MagicBlock's own blog names "liquidations, copy-trading and time-sensitive flows" as the target use cases. The feeds are delegated with "DelegateWithAnyValidator" and never auto-commit (commit_frequency_ms = u32::MAX), so they live permanently inside the ER where your delegated order-state accounts also live. Your ER program reads the price account and your order account in one ~10–50 ms ER transaction, updates trailing-stop high-water marks, and when triggered uses commit_and_undelegate / magic actions to settle on the base layer (where the Flash Trade execution leg would happen — Flash Trade's own programs are NOT inside the ER, so the fill must occur on mainnet L1 after commit). For trigger automation, MagicBlock also publishes "hydra" (permissionless Solana crank for scheduling instructions), or you run a thin keeper that fires ER transactions — either way the price comparison and trigger decision are fully on-chain inside the ER, which makes the ER integration load-bearing for eligibility. Caveat for scoping: the mainnet oracle is verifiably live but docs only list devnet endpoints as "currently free and supported for development", so the safe hackathon plan is devnet ER for the order engine demo (SOL/BTC/ETH/USDC feeds confirmed live there) with mainnet shown as stretch.
